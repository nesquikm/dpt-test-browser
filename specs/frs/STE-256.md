## Requirement

Define a platform-agnostic `WebviewAdapter` interface and ship a concrete `WebviewFlutterAdapter` built on `webview_flutter` ^4.13.1. The adapter is the sole surface through which the rest of the app drives an embedded webview — load URLs, navigate (back/forward/reload), and listen for load/title/error events. Also aligns platform deployment targets (iOS 13+, Android `minSdkVersion 24`, macOS 10.15+) so the implementation can boot on every M1 target. Per-tab ownership and UI wiring are downstream FRs.

## Acceptance Criteria

- AC-STE-256.1: `WebviewAdapter` abstract interface in `lib/browser/webview_adapter.dart` exposes methods `loadUrl(Uri)`, `goBack()`, `goForward()`, `reload()`, `dispose()`, and streams `onLoadStart`, `onLoadFinish`, `onTitleChanged`, `onLoadError`. A `WebviewLoadError` value type carries `Uri url`, `int? code`, `String message`.
- AC-STE-256.2: `WebviewFlutterAdapter` in `lib/browser/webview_flutter_adapter.dart` implements `WebviewAdapter`, wrapping a `WebViewController`. Constructor accepts an injectable `WebViewControllerFactory` typedef (default builds the real platform controller).
- AC-STE-256.3: `loadUrl`/`goBack`/`goForward`/`reload` delegate to the underlying `WebViewController`. Verified via a fake controller injected through the factory: calls reach the controller with identical arguments.
- AC-STE-256.4: webview_flutter `NavigationDelegate` callbacks fan out to streams — page-start→`onLoadStart`, page-finish→`onLoadFinish`, title-change→`onTitleChanged`, web-resource-error / HTTP-error→`onLoadError`.
- AC-STE-256.5: `dispose()` releases the underlying controller and closes all four streams. Calling any adapter method post-dispose is a no-op (does not throw).
- AC-STE-256.6: `FakeWebviewAdapter` under `test/fakes/fake_webview_adapter.dart` implements `WebviewAdapter`, exposing `emitLoadStart(Uri)`, `emitLoadFinish(Uri)`, `emitTitleChanged(String)`, `emitLoadError(WebviewLoadError)` so downstream cubit/widget tests drive deterministic event sequences.
- AC-STE-256.7: Platform deployment targets aligned — `ios/Podfile` `platform :ios, '13.0'`, `android/app/build.gradle.kts` `minSdkVersion 24`, `macos/Runner/Configs/AppInfo.xcconfig` `MACOSX_DEPLOYMENT_TARGET = 10.15`. macOS App Sandbox grants `com.apple.security.network.client` in both `macos/Runner/DebugProfile.entitlements` and `macos/Runner/Release.entitlements` so the embedded WKWebView can make outbound HTTP requests (Flutter's default templates omit the client entitlement; without it, every URL fails to load). `fvm flutter analyze` remains clean after edits.
- AC-STE-256.8: `technical-spec.md` §3 updated to list `onLoadError` + `dispose()` in the `WebviewAdapter` interface (closes the §3/§4 gap).
- AC-STE-256.9: Gate (`fvm flutter analyze && fvm flutter test`) is clean.

## Technical Design

- `lib/browser/webview_adapter.dart` — abstract `WebviewAdapter` class + `WebviewLoadError` Freezed value type.
- `lib/browser/webview_flutter_adapter.dart` — `WebviewFlutterAdapter` wrapping `WebViewController`. Ctor: `WebviewFlutterAdapter({WebViewControllerFactory? controllerFactory})`. `WebViewControllerFactory` typedef: `WebViewController Function()`.
- `test/fakes/fake_webview_adapter.dart` — `FakeWebviewAdapter` for cubit/widget tests downstream.
- `WebviewLoadError` fields source from `WebResourceError` (code, description) and `HttpResponseError` (status code) — absent fields fall back to safe defaults.
- Platform-target alignment touches `ios/Podfile`, `android/app/build.gradle.kts`, `macos/Runner/Configs/AppInfo.xcconfig` per webview_flutter 4.13.1's documented minimums.
- No new runtime deps beyond `webview_flutter` ^4.13.1 (already locked in technical-spec.md). Adds `webview_flutter` to `pubspec.yaml`.

## Testing

- `test/browser/webview_flutter_adapter_test.dart` — unit tests using a fake `WebViewController` via `WebViewControllerFactory`: delegation (`loadUrl`/`goBack`/`goForward`/`reload`), NavigationDelegate callbacks fan out to streams, `dispose()` closes streams + post-dispose calls are no-ops.
- `test/fakes/fake_webview_adapter_test.dart` — sanity: each `emit*` method produces a stream event.
- Real-platform behavior is verified manually per platform — `webview_flutter`'s platform internals are out of scope per testing-spec § What NOT to Test.

## Notes

- `onLoadError` is a separate stream rather than folded into `onLoadFinish` — keeps the happy-path subscription type clean and avoids a sentinel error payload (Socratic decision).
- Disposing a webview controller is async per `webview_flutter`; adapter `dispose()` awaits the platform controller close. Back-to-back close+open of the last tab is a known race (technical-spec §6) — exercised in the downstream Browser shell FR's widget tests, not here.
- macOS smoke ("loads google.com") is deferred — separate M1 task, not part of this FR.
- `webview_flutter` 4.13.1 does NOT expose `onHttpError` as a public field on `NavigationDelegate` — only the constructor parameter is wired through to the platform. The adapter therefore exposes a `@visibleForTesting void debugDispatchHttpError(HttpResponseError)` hook so tests can drive the same internal handler the platform invokes (mapping → `onLoadError` stream is identical either way). `onPageStarted` / `onPageFinished` / `onWebResourceError` are public fields and are tested by capturing the delegate and invoking them directly.
- Title-changed events: `webview_flutter` 4.x has no native title-changed callback. The adapter polls `controller.getTitle()` once per `onPageFinished` and emits on `onTitleChanged` only when the value actually changed — keeps the stream from firing redundantly on tabs whose title hasn't moved.
