## Requirement

Build the `BrowserShell` composite widget — tab strip on top, address bar (back / forward / reload + URL field) below, active tab's webview filling the rest. Wires user interactions to `TabManagerCubit` (open/close/select/navigate/setLoading/setTitle) and to the active tab's `WebviewAdapter` (loadUrl/goBack/goForward/reload). Each tab owns its own `WebviewAdapter`, created in the `TabView`'s `initState` and disposed when the tab leaves the tree. Load errors swap the webview area for an inline error UI with a Retry button.

This FR extends `STE-254`'s `TabManagerCubit` with `setTitle(String id, String title)` so `onTitleChanged` events have somewhere to land.

## Acceptance Criteria

- AC-STE-257.1: `BrowserShell` widget in `lib/ui/browser_shell.dart` is a `StatelessWidget` consuming `TabManagerCubit` via `BlocBuilder`. Top-down layout: `TabBar`, `AddressBar`, then the active tab's `TabView` filling remaining space. When `state.activeTabId == null`, shows an empty-state placeholder with a "New tab" button calling `cubit.openTab(kDefaultNewTabUrl)`.
- AC-STE-257.2: `TabBar` widget in `lib/ui/tab_bar.dart` renders `state.tabs` horizontally. Each tab shows: title (fallback to `url.host` when `title` is null), close (`×`) button, loading spinner when `tab.isLoading`. The active tab is visually distinguished. Tap → `cubit.selectTab(id)`. Close tap → `cubit.closeTab(id)`. Trailing `+` button → `cubit.openTab(kDefaultNewTabUrl)`.
- AC-STE-257.3: `AddressBar` widget in `lib/ui/address_bar.dart` renders the active tab's `url.toString()` in a `TextField`, plus back / forward / reload `IconButton`s. URL field commits on `onSubmitted` (Enter). Buttons call `adapter.goBack()` / `goForward()` / `reload()` on the active tab's adapter. All buttons are always enabled (no `canGoBack`/`canGoForward` in M1; out-of-history calls are webview-level no-ops).
- AC-STE-257.4: Address-bar input parsing lives in `lib/shared/url.dart` as `Uri? parseAddressBarInput(String input)`: `Uri.tryParse` first; if the result has an empty `scheme`, retry with `https://` prepended; on final failure return `null`. Null triggers a `SnackBar` "Invalid URL" via `ScaffoldMessenger.of(context)`; cubit state is unchanged. Successful parse → `cubit.navigate(activeId, parsedUri)` and `adapter.loadUrl(parsedUri)`.
- AC-STE-257.5: `TabView` widget in `lib/ui/tab_view.dart` is a `StatefulWidget` keyed by `tab.id`. `initState` builds a `WebviewAdapter` via an injectable `WebviewAdapterFactory` (default = `WebviewFlutterAdapter()`); `dispose` calls `adapter.dispose()`. Subscribes to: `onLoadStart` → `cubit.setLoading(id, true)` + `cubit.navigate(id, url)`; `onLoadFinish` → `cubit.setLoading(id, false)` + `cubit.navigate(id, url)` (covers in-page navigation); `onTitleChanged` → `cubit.setTitle(id, title)`.
- AC-STE-257.6: On `onLoadError`, `TabView` swaps the webview surface for an inline error widget showing the failed URL + error message + a Retry `TextButton`. Retry → `cubit.navigate(id, sameUrl)` → `adapter.loadUrl(sameUrl)`. The address bar still shows the failed URL so the user can edit + retry.
- AC-STE-257.7: Tab switch (`cubit.selectTab` changes `activeTabId`) re-renders the `BrowserShell` body so the new active tab's `TabView` (keyed by id) is shown. The previous active tab's `TabView` is removed from the tree → its `dispose` runs → its `WebviewAdapter.dispose()` runs. Re-selecting a previously-active tab builds a fresh `TabView` and reloads the URL (by design — no scroll/state preservation in M1).
- AC-STE-257.8: `TabManagerCubit.setTitle(String id, String title)` is added (extends STE-254): updates the matching tab's `title`. No-op if `id` is not present. STE-254's spec is patched with a new AC mirroring `setLoading`'s shape and a `bloc_test` is added under `test/tabs/tab_manager_cubit_test.dart`.
- AC-STE-257.9: Widget tests under `test/ui/` (`browser_shell_test.dart`, `tab_bar_test.dart`, `address_bar_test.dart`, `tab_view_test.dart`) use `FakeWebviewAdapter` (injected via the factory) to drive deterministic adapter events. Cover: tab strip click → `cubit.selectTab`; close button → `cubit.closeTab`; address-bar Enter (valid URL) → cubit.navigate + adapter.loadUrl; address-bar Enter (invalid) → snackbar shown, no cubit change; nav buttons → adapter delegation; `onLoadError` → inline error widget + Retry restores webview.
- AC-STE-257.10: Gate (`fvm flutter analyze && fvm flutter test`) is clean.

## Technical Design

- `lib/ui/browser_shell.dart` — top-level shell widget; `Scaffold` body with `Column(children: [TabBar, AddressBar, Expanded(child: TabView(key: ValueKey(activeId)))])`.
- `lib/ui/tab_bar.dart` — `Row` of tab chips inside a horizontally-scrolling `SingleChildScrollView`; trailing `+` button.
- `lib/ui/address_bar.dart` — `Row(children: [back, forward, reload, Expanded(TextField)])`; `TextEditingController` reset on active-tab change.
- `lib/ui/tab_view.dart` — owns the per-tab `WebviewAdapter`; ctor takes `WebviewAdapterFactory? factory` (default = real `WebviewFlutterAdapter`).
- `lib/shared/url.dart` — `parseAddressBarInput(String)` helper.
- `lib/shared/constants.dart` — `kDefaultNewTabUrl = Uri.parse('https://example.com')`.
- `WebviewAdapterFactory` typedef (`lib/browser/webview_adapter.dart`): `WebviewAdapter Function()`. Default factory: `() => WebviewFlutterAdapter()`. Test override: pass a closure returning `FakeWebviewAdapter`.
- New runtime deps: none (flutter_bloc, webview_flutter, freezed already present from STE-254/STE-256).

## Notes

- Tab switch recreates the `TabView` (keyed by id) → re-loads the page. UX trade-off accepted for M1; preserving scroll/state across tab switches is a later FR (would require adapter ownership in a registry or `IndexedStack` keep-alive).
- `kDefaultNewTabUrl` is `https://example.com` rather than `about:blank` because rendering of `about:blank` varies by platform in `webview_flutter` 4.x.
- `onTitleChanged` updates land on the cubit; tab strip re-renders. Because this extends STE-254, the cubit AC for `setTitle` is added as `AC-STE-254.9` when this FR is approved (STE-254 spec patch + Linear ticket update).
- Per-platform manual verification still required (macOS keyboard focus, iOS WKWebView cookie behaviour) per technical-spec §6.
- `parseAddressBarInput` rejects any input containing internal whitespace (after trim) — a real URL never has spaces, so this is the unambiguous "garbage" signal that AC-STE-257.4's "on final failure return null" rule needs to be deterministic. Without it, `Uri.tryParse('https://not a url')` percent-encodes the spaces and would silently accept it (underspecified backfill).
- UI class names are namespaced as `BrowserTabBar` (`lib/ui/tab_bar.dart`) and `BrowserTabView` (`lib/ui/tab_view.dart`) to avoid shadowing Material's own `TabBar` / `TabBarView` in callsite-local imports. Behavior unchanged from the AC text.
- The active tab's `WebviewAdapter` is owned by `BrowserTabView` (per AC-STE-257.5). `BrowserShell` bridges that adapter to `AddressBar` via a `ValueNotifier<WebviewAdapter?>` sink — `BrowserTabView.initState` publishes its adapter to the sink (post-frame, to avoid notifying listeners during build), and clears the sink on dispose (post-frame, to avoid notifying during teardown). `AddressBar` reads the sink via `ValueListenableBuilder` and renders a placeholder for the one frame between mount and publish.
