## Requirement

Replace the single-active `BrowserTabView` in `_ShellBody` with an `IndexedStack` over all open tabs so every tab's webview stays mounted across tab switches. Switching to a tab and back preserves DOM, scroll position, JS state, form input, and video playback — no reload, no URL refetch, no network round-trip.

This FR supersedes the M1 trade-off recorded in archived `AC-STE-257.7` ("re-selecting a previously-active tab builds a fresh `TabView` and reloads the URL — by design, no scroll/state preservation in M1"). The keep-alive policy is unconditional: no tab cap, no LRU eviction, no per-tab suspension. Adapter ownership stays inside `BrowserTabView.initState`/`dispose`; the address bar's adapter sink moves from "publish on view init" to "publish on activation" so only the active tab's adapter drives the back/forward/reload buttons.

## Acceptance Criteria

- AC-STE-260.1: `_ShellBody` in `lib/ui/browser_shell.dart` uses `IndexedStack` over `state.tabs`, with one `BrowserTabView` child per tab keyed by `ValueKey('tabview-${tab.id}')`. `index` is `state.tabs.indexWhere((t) => t.id == state.activeTabId)`. Children stay mounted across active-tab changes; their `WebviewAdapter`s are not disposed on switch.
- AC-STE-260.2: When the user switches from tab A to tab B and back to A, tab A's webview shows the same DOM, scroll position, JS state, form input, and video playback position that were present at switch-out. No reload event fires; no network request is issued.
- AC-STE-260.3: `cubit.closeTab(id)` removes that tab's `BrowserTabView` from the `IndexedStack`; its `dispose` runs and `WebviewAdapter.dispose()` runs exactly once. Closing an inactive tab does not dispose any other tab's adapter.
- AC-STE-260.4: The `AddressBar` adapter sink (`ValueNotifier<WebviewAdapter?>` on `BrowserShell`) is rebound to the active tab's adapter on every active-tab change. Back/forward/reload buttons and URL submission always operate on the active tab's adapter; switching tabs rebinds without an intermediate placeholder frame (after first-tab cold start).
- AC-STE-260.5: New-tab creation (`cubit.openTab`) appends a `BrowserTabView` to the `IndexedStack`; the adapter constructs and `loadUrl` fires immediately even when the new tab is not yet active. Background-loaded tabs surface load progress + title via existing cubit hooks.
- AC-STE-260.6: Widget tests in `test/ui/browser_shell_test.dart` cover: open 3 tabs → switch active across all 3 → all 3 `BrowserTabView`s stay in the widget tree (verified via `find.byType(BrowserTabView)` count); closing the inactive tab disposes only its `FakeWebviewAdapter`; address-bar adapter reference (read off the sink) matches the active tab's adapter after each `selectTab`.
- AC-STE-260.7: Manual platform smoke on macOS / iOS simulator / Android emulator: open 2 tabs (a media-heavy page + a scrolled page); play / scroll; switch tabs; switch back; verify state preserved with no black flash, no GPU surface re-attach artifact, no network request reissued in platform devtools / Charles. Result documented in this FR's Notes per platform.
- AC-STE-260.8: Gate `fvm flutter analyze && fvm flutter test` is clean.

## Technical Design

- `lib/ui/browser_shell.dart` — `_ShellBody` rebuilds: replace the single `Expanded(BrowserTabView(...))` with `Expanded(IndexedStack(index: activeIndex, children: [for (final t in state.tabs) BrowserTabView(key: ValueKey('tabview-${t.id}'), tab: t, ...)]))`. The `BlocBuilder<TabManagerCubit, TabManagerState>` already drives the rebuild on `state.tabs` / `state.activeTabId` changes.
- Adapter sink lifecycle moves out of `BrowserTabView.initState` / `dispose` and into `BrowserShell._BrowserShellState`. Each `BrowserTabView` still constructs/owns its adapter via `WebviewAdapterFactory`, but instead of publishing to the sink itself, it exposes its adapter via an `onAdapterReady(tabId, adapter)` callback into a shell-held `Map<String, WebviewAdapter>`; a separate `_activeAdapterSink` `ValueNotifier` tracks `map[activeTabId]` and updates whenever `state.activeTabId` changes or the map gains/loses an entry.
- `BrowserTabView` widget contract: keeps its existing constructor surface but the `adapterSink` parameter is replaced with `onAdapterReady` (tab-id-keyed). The post-frame publish/clear pattern still applies — publication happens after first build to avoid notifying listeners during build; clearing happens via post-frame in dispose. Per-tab clear is keyed by tab id so closing tab A does not clobber tab B's entry.
- `_EmptyState` path (no open tabs) is unchanged. Empty `state.tabs` → `IndexedStack(index: 0, children: const [])` would assert; the existing `state.activeTabId == null` early return in `BlocBuilder` continues to guard.
- No changes to `lib/tabs/`, `lib/browser/`, `lib/shared/`, or `lib/ui/tab_view.dart`'s adapter ownership rules — the change is localized to `_ShellBody` + the shell's adapter-tracking wiring.
- Risk: `webview_flutter` platform views (`AndroidView` / `UiKitView`) under an offstage `IndexedStack` child — historically, platform views have rendered as black or required a re-attach when their parent goes offstage and back onstage. Mitigation: AC-STE-260.7 makes per-platform manual smoke a hard requirement before merge. If a platform exhibits the artifact, fall back to `Stack + Offstage` per child (functionally equivalent but lets us add a per-platform repaint nudge in the offstage→onstage transition). The AC text does not change in either implementation path.

## Testing

- `test/ui/browser_shell_test.dart` — new group `'IndexedStack keep-alive'`:
  - Open 3 tabs via `cubit.openTab`; pump; `expect(find.byType(BrowserTabView), findsNWidgets(3))` regardless of which is active.
  - Switch active across each tab in turn; after each switch, all 3 `BrowserTabView`s remain in the tree, only one `FakeWebviewAdapter` instance per tab is constructed across the whole sequence (no adapter recreation on switch).
  - Close the second tab; pump; `expect(find.byType(BrowserTabView), findsNWidgets(2))`; the closed tab's `FakeWebviewAdapter.disposeCallCount` is 1; the others' are 0.
  - After each `selectTab`, the shell's `_activeAdapterSink.value` is identity-equal to the active tab's `FakeWebviewAdapter` (drives the AddressBar bridge contract).
- `test/ui/address_bar_test.dart` — extend to assert that after a tab switch, address-bar back/forward/reload taps land on the new active tab's adapter, not the previous one.
- `test/ui/tab_view_test.dart` — unchanged (per-tab init / dispose contract on `BrowserTabView` stays the same).
- Platform smoke (manual): documented in this FR's Notes, one row per platform. Required on macOS pre-merge; iOS / Android allowed as fast-follow if device access is gated.

## Notes

- Supersedes archived `AC-STE-257.7` ("re-selecting a previously-active tab builds a fresh TabView and reloads the URL — by design, no scroll/state preservation in M1"). The archive file is not edited; this Note is the canonical pointer.
- Memory policy: keep all open tabs alive simultaneously, no cap. Decision recorded during /brainstorm — explicit user trade-off ("Keep all alive, no cap"). If memory pressure becomes an observable issue (mobile OOM kills, sluggish UI on 20+ tabs), file a follow-up FR for LRU eviction or per-tab suspension; do not revisit this FR.
- Backgrounded tabs continue to run JS, timers, video playback, network requests. This is a deliberate consequence of the "full state preservation" choice — the alternative (pause-on-offstage) loses video position and breaks live-streaming pages.
- `webview_flutter` platform view behavior under `IndexedStack` is the principal implementation risk. Validate per-platform during /tdd before declaring AC-STE-260.2 GREEN; if a platform regresses, switch the implementation to `Stack + Offstage(...)` (still keeps children mounted) — the AC text does not change.
- **Implementation chose the Notes-authorised fallback layout**: a `Stack` of keyed `Visibility.maintain` wrappers, *not* a literal `IndexedStack` (deviation classification: `contradicts` — resolved per the Notes' "fall back to Stack + Offstage" clause). Reason: Flutter's `IndexedStack` wraps each child in an *unkeyed* `Visibility.maintain` internally, and when the children list shrinks (tab close), unkeyed positional reconciliation deactivates the right-most child even when it has a stable `ValueKey` — the closed tab's neighbours lose their `State` and adapters, breaking keep-alive. Wrapping each child in a *keyed* `Visibility.maintain` ourselves preserves key-based reconciliation across list shrinks. `Visibility.maintain` (rather than `Offstage` per the Notes' literal text) is preferred because it keeps the off-stage child laid out at full size, which is friendlier to `webview_flutter` platform views than `Offstage`'s size-zero collapse. The behavioural contract the AC asserts (children stay mounted, no dispose on switch, address-bar rebinds) is identical.
- **Test-discovery caveat**: under any keep-alive layout that hides off-stage children from `Element.debugVisitOnstageChildren` (literal `IndexedStack`, our `Stack + Visibility.maintain`, etc.), widget-test finders need `skipOffstage: false` to reach non-active children — `find.byType(BrowserTabView, skipOffstage: false)` and `find.byKey(..., skipOffstage: false)`. This is project-local convention captured in the test file under `test/ui/browser_shell_test.dart`.
